# DELIVERABLE 14: VENDOR & SUB-PROCESSOR REGISTRY

## Overview

DealStage relies on 16 third-party vendors to deliver its platform. Each vendor is assessed for data exposure, compliance posture, replacement difficulty, and risk level. This registry serves as the foundation for SOC-016 (Vendor Risk Management) and supports GDPR Article 28 sub-processor transparency requirements.

### Risk Level Key

| Level      | Criteria                                                                                                                  |
| ---------- | ------------------------------------------------------------------------------------------------------------------------- |
| **Low**    | No user PII processed; vendor failure causes minor degradation; easily replaceable                                        |
| **Medium** | Limited PII exposure or indirect data access; vendor failure causes feature degradation; replaceable with moderate effort |
| **High**   | Stores or processes significant user PII; vendor failure causes major outage; difficult to replace                        |

# DELIVERABLE 27: DATA RETENTION & PRIVACY MATRIX

### Legal Basis Key (GDPR Article 6)

| Basis                     | GDPR Reference | Meaning                                                                             |
| ------------------------- | -------------- | ----------------------------------------------------------------------------------- |
| **Consent**               | Art. 6(1)(a)   | Data subject has given explicit consent                                             |
| **Contractual Necessity** | Art. 6(1)(b)   | Processing is necessary for contract performance                                    |
| **Legal Obligation**      | Art. 6(1)(c)   | Processing is required by law                                                       |
| **Legitimate Interest**   | Art. 6(1)(f)   | Processing serves legitimate business interest balanced against data subject rights |

Architecture Decision Records (ADRs) document the rationale behind significant technical decisions in the DealStage platform. Each ADR captures the context, alternatives considered, trade-offs evaluated, and consequences -- both positive and negative -- of the chosen approach. ADRs are immutable once accepted; if a decision is reversed, a new ADR is created that supersedes the original.

### ADR-001: Supabase Over Firebase or Custom Backend

| Field                       | Detail                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| --------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **ADR ID**                  | ADR-001                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                |
| **Title**                   | Use Supabase as the Backend-as-a-Service Platform                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| **Status**                  | Accepted                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                               |
| **Date**                    | 2025-10-01                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                             |
| **Context**                 | DealStage needed a backend that could support relational data (brands, talents, partnerships with complex relationships), authentication, real-time subscriptions, and serverless functions. The platform was being built by a solo founder who needed to move fast without managing infrastructure. The data model requires complex queries (JOINs across brands, talents, decision makers, partnerships) and row-level security for a multi-tenant SaaS application with 5 distinct user roles.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| **Decision**                | Use Supabase (PostgreSQL + Auth + Edge Functions + Storage + Realtime) as the single backend platform.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| **Alternatives Considered** | **Firebase (Firestore + Functions):** NoSQL document model is a poor fit for the relational data model (brands-to-talents many-to-many, partnership pipeline with status history). Firestore queries cannot do JOINs, requiring denormalization that increases complexity and inconsistency risk. Firebase Auth is comparable. Cloud Functions have cold start issues. **Custom Backend (Express/Fastify + PostgreSQL + Auth0):** Maximum flexibility but massive operational overhead for a solo founder. Requires managing hosting, database backups, auth integration, API development, and deployment pipeline. Estimated 3-4x longer time-to-market. **Hasura + PostgreSQL:** GraphQL-first approach. Good for complex queries but adds a layer of abstraction. Less mature auth integration. Smaller community for troubleshooting. **PlanetScale + Clerk + Vercel Functions:** Distributed MySQL with separate auth. MySQL lacks PostgreSQL features (JSONB, array types, RLS) that DealStage heavily uses for flexible schema (demographics jsonb, audience_interests text[]). |
| **Consequences**            | **Positive:** (1) PostgreSQL provides the relational model needed for complex brand-talent relationships with proper JOINs and referential integrity. (2) RLS eliminates an entire class of authorization bugs -- security is enforced at the database layer, not just the API layer. (3) Edge Functions (Deno) run close to the database, minimizing latency for database-heavy operations. (4) Built-in Auth with JWT integrates seamlessly with RLS via `auth.uid()`. (5) Single vendor reduces operational complexity. (6) Generous free tier during development. **Negative:** (1) Vendor lock-in -- migration from Supabase requires rewriting all 100+ Edge Functions, RLS policies, and auth logic (estimated 6+ months). (2) Edge Functions are Deno-based, limiting npm package compatibility (though improving). (3) Supabase is younger than Firebase with a smaller ecosystem. (4) Single region (us-east-1) creates latency for non-US users. (5) Edge Function cold starts affect first-request latency.                                                                |
| **Related ADRs**            | ADR-008 (React + Vite), ADR-009 (Vercel), ADR-012 (Edge Functions)                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                     |

### ADR-002: SECURITY DEFINER RPC Over Direct Profile Queries

| Field                       | Detail                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| --------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **ADR ID**                  | ADR-002                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| **Title**                   | Use SECURITY DEFINER RPC for Profile Loading Instead of Direct Table Queries                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                |
| **Status**                  | Accepted                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                    |
| **Date**                    | 2025-12-15                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                  |
| **Context**                 | During development, a critical race condition was discovered: when a user logs in via OAuth, the Supabase client issues a profile query immediately after receiving the JWT. However, PostgreSQL's RLS evaluation of `auth.uid()` can fail during the brief window where the JWT is being processed but the auth context is not yet fully established in the database session. This caused intermittent "permission denied" errors on the `profiles` table, particularly for first-time OAuth users where the `handle_new_user` trigger had not yet completed. The issue was most visible on Google OAuth login where the redirect + session establishment + profile query happen in rapid succession.                                                                                                                      |
| **Decision**                | Create a `get_my_profile` PostgreSQL function with `SECURITY DEFINER` (runs as the function owner, bypassing RLS) that accepts no parameters and internally uses `auth.uid()` to fetch the authenticated user's profile. All profile loading in `AuthContext.jsx` routes through this RPC instead of direct `supabase.from('profiles').select()`.                                                                                                                                                                                                                                                                                                                                                                                                                                                                           |
| **Alternatives Considered** | **Add retry logic to direct query:** Would mask the race condition with exponential backoff. Fragile -- timing-dependent. Adds latency to every login. Doesn't solve the root cause. **Modify RLS policy to be more permissive:** Weakens security model to work around a timing issue. Unacceptable trade-off. **Use Supabase service_role key client-side:** Catastrophic security anti-pattern -- service_role bypasses all RLS. Never expose service_role key to the browser. **Add delay between auth and query:** Hacky. Unreliable across different network conditions. Poor UX.                                                                                                                                                                                                                                     |
| **Consequences**            | **Positive:** (1) Eliminates the race condition entirely -- function runs with elevated privileges. (2) Zero-parameter design prevents injection -- the function internally resolves the caller via `auth.uid()`. (3) Single point of control for profile loading logic. (4) No additional latency -- single RPC call vs. SELECT. (5) Pattern documented in Supabase official docs for this exact scenario. **Negative:** (1) SECURITY DEFINER functions bypass RLS, so bugs in the function body could leak data. Mitigated by: function only returns the authenticated user's own row, no parameters to manipulate. (2) Developers must remember to use the RPC, not direct queries. Enforced by code convention documented in AuthContext. (3) Function must be maintained in sync with `profiles` table schema changes. |
| **Related ADRs**            | ADR-001 (Supabase), ADR-004 (Role Parity)                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                   |

### ADR-003: Dual AI Model Strategy (Claude + GPT-4o-mini)

(4) Post-trial downgrade can feel punitive -- "I was doing X and now I can't." Mitigated: clear in-app messaging about which features require upgrade. |

### ADR-011: Single Source of Truth Pattern for Plans and Talent Types

| Field                       | Detail                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                    |
| --------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **ADR ID**                  | ADR-011                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                   |
| **Title**                   | Maintain Single Source of Truth Files for Configuration Data                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                              |
| **Status**                  | Accepted                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                  |
| **Date**                    | 2026-03-15                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                |
| **Context**                 | DealStage has multiple configuration datasets that are referenced across the codebase: (1) page access rules (which roles/tiers can access which pages), (2) talent type definitions (140+ types across 12 categories), (3) feature flags, (4) plan definitions. Early in development, this data was duplicated across multiple files -- `pageAccess.js`, `featureFlags.js`, navigation config, and individual page components each had their own access checks. This led to: conflicting access rules (page accessible in nav but blocked by guard), stale feature flag definitions, and talent type lists that differed between search filters and onboarding dropdowns.                                                                                                                                                                                                |
| **Decision**                | Establish Single Source of Truth (SSOT) files: (1) `src/config/pageAccess.js` -- all page access rules, feature flags, and tier definitions. (2) `src/lib/talentTypes.js` -- all talent type definitions. Other files that previously contained duplicated configuration (e.g., `src/lib/featureFlags.js`) are refactored to import from and delegate to the SSOT files. The pattern is: define once, derive everywhere.                                                                                                                                                                                                                                                                                                                                                                                                                                                  |
| **Alternatives Considered** | **Database-driven configuration:** Store page access rules and talent types in Supabase tables (`subscription_plans` TBL-055, `talent_types` TBL-042). Allows runtime changes without deployment. However: adds latency to every page load (must query config before rendering), increases Supabase API usage, and makes the access control system dependent on database availability. Configuration changes still need testing -- "hot" config changes are dangerous for access control. **Environment variables:** Store tier/feature definitions in Vercel environment variables. Inappropriate for complex nested data structures. No type safety. Hard to version control. **Separate config service:** Overkill for current scale. Adds infrastructure complexity.                                                                                                  |
| **Consequences**            | **Positive:** (1) One place to edit, guaranteed consistency across the entire app. (2) `featureFlags.js` delegates to `pageAccess.js` -- old API surface preserved, new callers use SSOT directly. (3) Compile-time configuration -- no runtime database dependency for access control. (4) Git-tracked -- every configuration change has a commit history with author and date. (5) Easy to audit: "show me all access rules" = read one file. **Negative:** (1) Configuration changes require a deployment (no hot reloading). Acceptable trade-off for access control where changes should be deliberate and tested. (2) Database tables (`subscription_plans`, `talent_types`) may drift from SSOT files if not synchronized. Mitigated: seed scripts reference the same data. (3) Large config files can become unwieldy -- `pageAccess.js` already has 80+ entries. |
| **Related ADRs**            | ADR-004 (Role Parity implemented via pageAccess.js SSOT), ADR-008 (Vite enables fast reloads of config changes)                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                           |

### ADR-012: Supabase Edge Functions Over Serverless (Lambda/Cloud Functions)

| Field                       | Detail                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                            |
| --------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **ADR ID**                  | ADR-012                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                           |
| **Title**                   | Use Supabase Edge Functions (Deno) Over AWS Lambda or Google Cloud Functions                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| **Status**                  | Accepted                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                          |
| **Date**                    | 2025-10-15                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                        |
| **Context**                 | DealStage requires server-side logic for: AI API calls (protecting API keys), Stripe webhook handling, data enrichment orchestration, email sending, user account management, and 30+ AI agent implementations. This logic cannot run in the browser (API key exposure) and must interact heavily with the Supabase database (reading profiles, writing AI logs, updating subscriptions). The platform needs 100+ distinct server-side functions.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| **Decision**                | Use Supabase Edge Functions (Deno runtime) for all server-side logic. Functions are deployed via Supabase CLI and run on Deno Deploy's edge network. Functions have direct access to Supabase's service_role key for RLS bypass when needed. Shared utility code stored in `supabase/_shared/`.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                   |
| **Alternatives Considered** | **AWS Lambda (Node.js):** Industry standard serverless. Maximum flexibility and ecosystem. However: (1) separate AWS account and IAM configuration, (2) must manage Supabase connection from external network (connection pooling, SSL), (3) cold starts on Node.js Lambda are 200-500ms, (4) CloudWatch logging adds complexity, (5) API Gateway needed for HTTP endpoints. Total infrastructure overhead is significantly higher. **Google Cloud Functions:** Similar trade-offs to Lambda. Google Cloud ecosystem is less familiar. **Vercel Serverless Functions:** Already on Vercel platform. However: (1) Vercel functions are designed for Next.js API routes, not standalone serverless, (2) would create split between Vercel functions (HTTP) and Supabase Edge Functions (database-heavy), (3) Vercel function execution limits may be restrictive for AI agent tasks. **Cloudflare Workers:** Fastest cold start in the industry. However: (1) V8 isolate model has strict CPU time limits (50ms free, 30s paid), unsuitable for AI API calls that can take 5-30 seconds, (2) no native Supabase integration.                                                                                        |
| **Consequences**            | **Positive:** (1) Co-located with database -- Edge Functions run on the same infrastructure as Supabase, minimizing network latency for database queries. (2) Automatic access to Supabase service_role and anon keys via environment variables. (3) `supabase.auth.getUser()` available in every function for authentication. (4) Deno's security model (explicit permissions) is a good fit for server-side functions. (5) 100+ functions deployed and managed through a single `supabase/functions/` directory. (6) No additional billing -- included in Supabase Pro plan (with invocation limits). **Negative:** (1) Deno runtime -- not all npm packages are compatible (though `npm:` specifiers help). (2) Edge Functions have a 60-second execution limit -- long-running AI tasks must be designed around this. (3) Cold starts on first invocation after idle period (100-500ms). (4) Debugging is harder than local Node.js -- requires `supabase functions serve` for local development. (5) Deployment via CLI is manual (not yet in CI pipeline -- see SOC-012). (6) No built-in job queue or scheduled execution -- cron-like features require external triggers or Supabase's pg_cron extension. |
| **Related ADRs**            | ADR-001 (Supabase as backend platform), ADR-003 (AI providers called from Edge Functions), ADR-007 (Stripe webhook handled by Edge Function)                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
